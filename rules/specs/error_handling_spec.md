# 错误处理规范

> 触发条件：涉及外设失败、通信失败、传感器离线、参数非法、错误提示、降级运行、复位策略或看门狗配合时，必须先阅读本文件。

错误处理必须帮助现场排障和系统恢复，不应只是让编译通过。具体错误码、日志接口和复位策略必须匹配当前工程。

## 开发前必须确认

- 当前工程是否已有错误码、返回值约定、日志模块、告警模块和复位诊断。
- 外设驱动返回错误的方式：返回码、状态位、回调、全局错误寄存器或异常中断。
- 哪些模块是关键功能，哪些模块可以离线降级。
- 是否启用看门狗、掉电保存、复位原因记录或远程诊断。
- 错误信息最终输出到哪里：调试串口、显示屏、LED、蜂鸣器、日志存储、上位机或云端。

## 错误分类原则

- 参数错误：空指针、长度越界、非法枚举、单位不匹配、范围不合法。
- 超时错误：等待硬件、对端、采样完成、发送完成或确认响应超时。
- 通信错误：总线错误、校验失败、对端无响应、协议不合法或链路中断。
- 数据错误：传感器读数越界、标定缺失、滤波输入异常或状态不可信。
- 存储错误：Flash、EEPROM、外部存储或参数保存失败。
- 严重错误：关键外设持续不可用、控制输出无法保证安全、系统状态不可恢复。

## 处理原则

- 错误必须显式处理，不允许静默忽略。
- 错误信息应包含足够定位根因的上下文，例如模块、操作、资源、参数范围、返回状态和失败次数。
- 单次普通错误不应立即让整机停摆；应记录并按模块策略重试、保留上次可信值或进入受控降级。
- 连续失败必须有独立计数，达到阈值后进入降级、告警或严重错误路径。
- 成功恢复后应清除连续失败状态，但保留必要的累计诊断信息。
- 严重错误不能用原地死循环掩盖；是否等待看门狗复位必须符合项目安全策略。

## 降级运行原则

- 传感器离线时，显示或上报离线状态，不伪造正常数据。
- 通信失败时，本地控制和安全逻辑应尽量继续运行，通信模块按节流策略恢复。
- 显示失败时，核心采集、控制和通信不应被显示模块拖死。
- 存储失败时，使用明确的默认参数或上次有效参数，并提示参数未保存。
- 执行机构控制异常时，优先进入安全状态，并记录触发原因。

## 复位与看门狗

- 启用看门狗的项目，应在启动早期读取并记录复位原因，再清除复位标志。
- 复位原因记录必须放在当前芯片和 SDK 支持的接口上实现，不使用其他厂商宏名。
- 连续看门狗复位需要进入更强告警或保护策略，避免无限重启而无诊断信息。
- 没有看门狗的项目，严重错误也必须有可观察状态，不能停在不可诊断的死循环。

## 禁止事项

- 禁止 catch-all 式吞掉所有错误。
- 禁止用固定错误码表覆盖当前工程已有错误体系。
- 禁止失败后只返回通用失败值而不记录模块和操作上下文。
- 禁止把断言当作运行时错误处理。
- 禁止添加隐藏根因的 fallback，除非用户明确要求。

## 完成检查清单

- 所有外设调用、协议解析、参数入口和缓冲区操作都有失败路径。
- 错误上下文足够复现或定位问题。
- 连续失败、恢复成功和降级退出条件明确。
- 看门狗与严重错误策略一致，不掩盖卡死。
- 错误处理接口匹配当前工程风格。

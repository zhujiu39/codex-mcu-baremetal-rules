# 显示界面规范

> 触发条件：涉及 OLED、LCD、数码管、LED 阵列、状态灯、菜单、页面、阈值设置、设备控制、显示刷新或长文本显示时，必须先阅读本文件。

显示规则必须服务当前硬件和产品交互，不固定屏幕尺寸、页面数量、按键布局或视觉风格。

## 开发前必须确认

- 显示器件类型、分辨率、颜色能力、刷新接口、驱动芯片和显存模型。
- 当前工程已有显示驱动、字体库、绘图缓冲、刷新函数和 UI 模块。
- 显示内容：监测数据、告警、参数设置、设备控制、调试信息或启动状态。
- 输入方式：按键、旋钮、触摸、串口命令、远程配置或无交互只显示。
- 刷新频率、通信带宽、CPU 占用、功耗和低功耗唤醒限制。

## 架构原则

- 显示驱动负责像素、字符和刷新；UI 层负责页面状态和交互；业务层提供数据，不直接绘制。
- 主循环文件只调用 UI 或显示任务入口，不直接拼接页面细节。
- UI 层读取业务状态时应使用稳定快照，避免显示一半旧数据一半新数据。
- 显示刷新必须可调度、可节流，不用阻塞延时控制节奏。
- 绘制和硬件刷新应分离；若硬件或驱动支持显存缓冲，优先使用缓冲后统一刷新。

## 内容设计原则

- 页面数量和信息层级按真实屏幕尺寸、输入方式和用户任务确定。
- 数据必须显示单位、状态和异常含义，避免让无效值看起来像正常值。
- 阈值、模式和控制状态应有明确选中态、编辑态和保存态。
- 文本、数字和图标不得互相覆盖；超长内容必须截断、滚动、分页或缩写。
- 告警信息应可被用户发现，但不能频繁刷屏影响核心任务。

## 刷新与性能

- 刷新周期由屏幕特性、数据变化速度、总线带宽和功耗决定，不写跨项目固定值。
- 高频变化数据应限制显示刷新频率，避免显示任务占用主循环。
- 输入事件、告警变化和关键状态变化可以触发即时刷新，但仍需避免连续抖动刷新。
- 硬件刷新失败必须有错误处理，不应拖死采集、控制或通信任务。

## 禁止事项

- 禁止固定某一种屏幕尺寸、页面布局或按键含义作为通用规则。
- 禁止在中断中绘制 UI 或刷新硬件屏幕。
- 禁止在绘制过程中执行传感器采样、通信等待或控制动作。
- 禁止为了显示美观引入阻塞动画或长时间占用总线。
- 禁止显示未校验或已失效的数据而不标记状态。

## 完成检查清单

- 已确认显示硬件、驱动接口、缓冲模型和刷新限制。
- UI 层、显示驱动和业务模块职责分离。
- 刷新策略不会阻塞主循环或破坏通信时序。
- 输入事件、告警和远程配置对 UI 的影响明确。
- 小屏幕文本和数值溢出已有处理策略。
